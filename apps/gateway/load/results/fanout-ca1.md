# FEN-50 — F7/CA1 gateway fan-out load proof

**Verdict: PASS ✅** — 1 000 concurrent sockets each receive exactly one DELTA
frame per pixel write, with zero per-socket DB read on the fan-out path and a
single Redis delta subscription regardless of socket count.

Generated by `apps/gateway/load/fanout-load.ts` (run: `LOAD_PLATEAUS=1,10,50,100,250,500,1000 LOAD_ROUNDS=5`).
Gateway under test: `apps/gateway` in the `_default` canonical tree (F7/FEN-13 fan-out,
now with F4/F5/F8 placement+moderation wired — the fan-out path is unchanged).

## CA1 invariants

| Invariant | Result |
|---|---|
| Every socket receives exactly one DELTA per write (no drops, no dups), carrying that write's `seq` | ✅ `everySocketOneDelta` |
| Zero DB/Redis read or write triggered **per socket** on the fan-out path | ✅ `zeroPerSocketDbRead` |
| Zero Convex function calls on the read/fan-out path | ✅ by construction (StaticGaugeBonusSource; no `CONVEX_URL`) |
| Exactly one Redis delta subscription regardless of socket count (fan-out is in-process) | ✅ `oneSubscription` |
| No dropped frames as the count climbs 1 → 1 000 | ✅ `noDroppedFrames` |
| Fan-out p99 within ceiling (150 ms) | ✅ `latencyWithinCeiling` |

- **Δ Redis cmds on fan-out** = total Redis commands issued by the gateway in the
  window where one pixel delta is published and fanned out to all N sockets. It is
  **0 at every N** — the broadcast is a pure in-process loop over the client set
  (`Gateway.flush`), reading nothing per socket. This is the core CA1 claim.
- **firstByte** = publish → first client receives (≈ the 10 ms coalescer flush
  interval + dispatch). **spread** = first client → last client receives, i.e. the
  server-side cost of serializing one pre-encoded frame to N sockets. At 1 000
  sockets the whole broadcast spreads over only **~16 ms** — sub-linear and tiny.
- p50/p95/p99 are end-to-end (publish → client decode). They grow modestly because
  this harness co-locates all 1 000 *clients* with the 1 000 *server* sockets in a
  single Node process on 4 shared cores; client-side receive scheduling inflates
  the figure a real distributed deployment would not pay. The isolated **spread**
  metric is the honest server-side number.
- Memory grows ~36 MB across 0 → 1 000 sockets (~36 KB/socket of buffers + state).

## Method & fidelity

- **Real** WebSocket server + sockets, **real** `Gateway`, `DeltaCoalescer`,
  `SeqRingBuffer`, encode/decode, and in-process fan-out loop — the production code
  paths. Each synthetic client connects, consumes welcome+snapshot, then asserts it
  observes exactly one DELTA per published write at the expected `seq`.
- Redis is substituted by an **instrumented in-process double**
  (`load/instrumentedRedis.ts`) that counts every command by name and handle. This
  is a *stronger* proof of "command count stays flat" than a live server: the exact
  command stream is observed with no sampling, and the structural invariants
  (one subscription, zero per-socket reads) are impl-level facts a real Redis would
  not change.
- Background presence/heartbeat timers are pushed past the test window so the
  per-delta command diff isolates the fan-out path (presence is O(1)/instance, not
  per-socket, regardless).

## Caveat / recommended follow-up

This proves the CA1 **architecture** (in-process fan-out, single subscription, zero
per-socket DB read) and fan-out behaviour under 1 000 sockets. It does **not** cover
a live Redis pub/sub hop or distributed client RTT on real hardware — that belongs to
the **NAS bring-up + smoke (FEN-25)**, where the same assertion (place one pixel →
every connected client sees one DELTA with matching `seq`; Redis `INFO` command count
flat vs. socket count) should be re-run against the deployed stack. No product code
change was required; this is verification scaffolding only.
